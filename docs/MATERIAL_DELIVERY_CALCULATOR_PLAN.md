# План реалізації: калькулятор доставки матеріалів + єдина логіка розрахунку для запланованих і термінових заявок

> Документ для AI-агента Codex, який уже знаходиться в контексті проєкту TechnoRent.
> Мета: реалізувати нову послугу доставки матеріалів, не створюючи окремий контур, а вбудовуючи її в існуючу архітектуру `Service -> CustomerRequest -> RentOrder -> WorkAssignment / Telegram / GPS / Close`.

## Поточний стан реалізації

- Реалізовано публічний калькулятор доставки матеріалів на сторінці послуги.
- Реалізовано режими `Чим швидше` і `На запланований час`.
- Реалізовано snapshot заявки з повними деталями доставки матеріалів у `CustomerRequest`.
- Реалізовано відображення деталей доставки матеріалів в адмінці (`Заявки`, `Замовлення`).
- Реалізовано відображення точок постачання на вкладці `Мапа`.
- Реалізовано передачу точок завдання працівнику в Telegram flow.
- Виправлено критичний backend-баг: якщо координати адреси не були передані з фронтенду, `null` перетворювався в `0,0`, через що розрахунок падав із повідомленням про неможливість побудувати маршрут.
- Поточний live-стан: локальний розрахунок для `Львів, вул. Городоцька, 120` знову працює і в `urgent`, і в `scheduled`.

## 1. Мета змін

Потрібно реалізувати нову послугу **доставки матеріалів** так, щоб вона:

- була частиною вже існуючого блоку послуг;
- використовувала той самий CRM-контур, що й інші звернення;
- створювала заявки у `CustomerRequest`, а не в окрему нову систему;
- за потреби переводилась у `RentOrder`;
- використовувала існуючі GPS, карту, техніку, працівників і Telegram flow;
- не дублювала вкладки, сторінки та бізнес-логіку.

Одночасно потрібно **узгодити логіку розрахунку** для двох калькуляторних послуг:

1. `tow_calculator` — евакуатор;
2. новий `material_delivery_calculator` — доставка матеріалів.

Ключова продуктова ідея:

- для **термінового** сценарію розрахунок виконується від **поточної GPS-позиції** техніки;
- для **запланованого** сценарію розрахунок виконується від **бази техніки**, а не від live GPS;
- користувач сам явно обирає режим у модальному вікні калькулятора через перемикач:
  - `Чим швидше`;
  - `На запланований час`.

## 2. Принципи реалізації

### 2.1. Не створювати окремий контур

Усе повинно бути вбудовано в поточну систему:

- послуга зберігається в `Service`;
- публічний розрахунок працює через сторінку послуги та калькулятор;
- заявка потрапляє в єдину вкладку `Заявки`;
- менеджер переводить звернення в `RentOrder` через вже існуючу логіку;
- виконання йде через існуючий execution flow.

### 2.2. Не показувати зайву бізнес-інформацію клієнту

Клієнт **не повинен бачити**:

- постачальника;
- назву точки завантаження;
- альтернативні точки;
- деталі внутрішньої логіки вибору.

Клієнт бачить тільки:

- вартість доставки;
- вартість матеріалу;
- орієнтовну загальну суму;
- пояснення, що це попередній розрахунок.

### 2.3. Адмін має бачити повну картину

Менеджер в адмінці має бачити:

- яку точку система вибрала;
- чому саме її;
- альтернативні точки;
- розбивку по сумі;
- режим розрахунку (`urgent/live` або `scheduled/base`);
- збережений snapshot усіх параметрів на момент заявки.

### 2.4. Не ламати існуючу логіку

Не можна:

- повертати Prisma в runtime;
- виносити секрети у frontend;
- створювати окрему вкладку заявок на доставку матеріалів;
- зламати `tow_calculator`;
- зламати `CustomerRequest`, `RentOrder`, `BookedPeriod`, `GPS`, `Map`, Telegram flow;
- робити нові Postgres enum там, де це можна вирішити `TEXT + backend validation`.

## 3. Нове продуктове рішення

### 3.1. Новий тип ціни для послуги

Потрібно додати новий `pricingType`:

```ts
material_delivery_calculator
```

Він означає, що послуга:

- має окремий калькулятор;
- розраховує вартість матеріалу + вартість доставки;
- використовує точки постачання;
- вибирає найвигіднішу точку автоматично;
- може використовувати live GPS або базу техніки залежно від режиму.

### 3.2. Нове поле у послузі

Потрібно додати до `Service` нове поле:

```ts
deliveryRatePerKm: number | null
```

Призначення:

- числовий тариф доставки за 1 км;
- використовується в обчисленнях;
- має бути окремим від `priceInfo`.

### 3.3. Роль `priceInfo`

`priceInfo` не повинен бути джерелом істини для обчислень.

Його роль:

- маркетинговий або пояснювальний текст на сайті;
- візуальний опис формату ціноутворення.

Джерело істини для розрахунків:

- `deliveryRatePerKm`;
- ціна матеріалу в точці постачання;
- відстані по маршрутах.

### 3.4. Узгоджена логіка для обох калькуляторів

Потрібно не робити дві різні концепції, а запровадити **єдине правило**:

#### Режим 1: `Чим швидше`

- розрахунок іде від **поточної GPS-позиції техніки**;
- використовується для термінових заявок;
- логіка актуальна для:
  - евакуатора;
  - доставки матеріалів.

#### Режим 2: `На запланований час`

- користувач вводить дату / час;
- розрахунок іде **не від live GPS**, а від **бази техніки**;
- використовується для майбутніх заявок;
- логіка однакова для:
  - евакуатора;
  - доставки матеріалів.

## 5. Алгоритм розрахунку

## 5.1. Для доставки матеріалів

Для кожної точки, де є потрібний матеріал, треба порахувати:

### 1. Вартість матеріалу

```text
materialCost = materialUnitPriceAtPoint * requestedQuantity
```

### 2. Вартість доставки

```text
deliveryCost = deliveryRatePerKm * (truckToLoadingPointKm + loadingPointToClientKm)
```

### 3. Загальна вартість

```text
totalCost = materialCost + deliveryCost
```

### 4. Вибір найкращої точки

Система повинна автоматично вибрати **найвигіднішу точку**:

- матеріал доступний;
- точка активна;
- є валідна ціна;
- є валідні координати;
- загальна сума `totalCost` мінімальна.

### 5. Ключова різниця режимів

#### Якщо `Чим швидше`

```text
truckToLoadingPointKm = distance(currentGpsPosition -> supplierPoint)
```

#### Якщо `На запланований час`

```text
truckToLoadingPointKm = distance(equipmentBaseLocation -> supplierPoint)
```

У будь-якому разі:

```text
loadingPointToClientKm = distance(supplierPoint -> clientAddress)
```

---

## 5.2. Для евакуатора

Логіка та сама, тільки без матеріалу.

### Якщо `Чим швидше`

Розрахунок іде від:

- поточної GPS-позиції евакуатора.

### Якщо `На запланований час`

Розрахунок іде від:

- бази евакуатора.

### Формула

```text
totalCost = deliveryRatePerKm * (startPointToPickupKm + pickupToDropoffKm)
```

де:

- `startPointToPickupKm`:
  - live GPS у режимі `Чим швидше`;
  - база в режимі `На запланований час`.

## 6. Нові дані та сутності

## 6.1. База техніки

Щоб коректно рахувати заплановані заявки, техніка, яка бере участь у калькуляторах, повинна мати **базову локацію**.

Мінімально потрібно додати для техніки:

- `baseAddress`;
- `baseLatitude`;
- `baseLongitude`.

Це поле потрібне для:

- евакуатора;
- самоскидів / вантажних авто;
- іншої техніки, яка бере участь у калькуляторних послугах.

> База техніки — це не поточне GPS-розташування, а стабільна стартова точка для запланованих заявок.

## 6.5. Snapshot розрахунку в CRM

Під час створення заявки потрібно зберігати **не тільки поля форми**, а й **snapshot розрахунку**.

Мінімум треба зберігати:

- `calculationMode` = `urgent_live` | `scheduled_base`
- `servicePricingType`
- `selectedMaterialId?`
- `selectedMaterialName?`
- `quantity?`
- `unit?`
- `deliveryRatePerKm`
- `materialCost`
- `deliveryCost`
- `totalEstimatedCost`
- `truckToPointKm?`
- `pointToClientKm?`
- `chosenSupplierPointId?`
- `chosenSupplierPointName?`
- `chosenOfferUnitPrice?`
- `alternativesSnapshot?`
- `calculatedAt`

### Важливо

- клієнту не показуємо `chosenSupplierPointName` і `alternativesSnapshot`;
- в адмінці ці поля показуємо.

Формат збереження:

- бажано `JSONB`, якщо це вже використовується в системі;
- якщо поточна модель не дозволяє це зробити акуратно — дозволяється зберігати як serialized JSON у `TEXT`, але з чітким backend-parse / validate.

## 8.5. Інтеграція зі сторінками послуг

Для послуг із `pricingType = material_delivery_calculator` сторінка послуги має відкривати саме новий калькулятор.

Для `pricingType = tow_calculator` — оновлений tow calculator.

Логіка відкриття повинна залишатись у поточній архітектурі сторінок послуг, а не через сторонній окремий маршрут.

---

## 9. Зміни в адмінці

## 9.1. Єдина вкладка `Заявки`

Заявки на доставку матеріалів повинні потрапляти в ту саму вкладку `Заявки`.

У деталях такої заявки менеджер повинен бачити:

- матеріал;
- кількість;
- одиницю виміру;
- режим розрахунку;
- доставку;
- матеріал;
- загальну суму;
- обрану системою точку;
- альтернативні точки;
- розбивку вартості;
- дату/час;
- коментар;
- кнопку `Створити замовлення`.

---

## 9.2. Створення замовлення з заявки

Коли менеджер переводить таку заявку в `RentOrder`, потрібно перенести snapshot і ключові поля:

- адреса клієнта;
- точка завантаження;
- матеріал;
- кількість;
- погоджена або попередня ціна;
- режим розрахунку;
- пов’язану техніку для доставки.

Потрібно зберегти сумісність з уже існуючою логікою `RentOrder`.

---

## 9.3. Нова секція в адмінці

Потрібно додати новий розділ у sidebar, наприклад:

### `Постачання`

Усередині нього:

1. `Матеріали`
2. `Точки постачання`
3. `Пропозиції / ціни`

Це повинен бути **довідниковий контур**, а не новий контур заявок.

---

## 9.4. Розширення мапи

Потрібно розширити наявну карту в адмінці, щоб вона могла показувати:

- GPS техніки;
- точки постачання;
- за потреби обидва шари разом.

Для точок постачання показувати:

- назву;
- активність;
- доступні матеріали;
- ціни;
- дату оновлення;
- перехід у редагування.

Це можна робити окремим фільтром або перемикачем шарів на поточній сторінці карти.

### Результат етапу

Короткий технічний опис:

## 11. Критерії приймання

Результат вважається правильним, якщо:

1. Додано новий `pricingType = material_delivery_calculator`.
2. У послугах можна задати `deliveryRatePerKm`.
3. У техніки є базова локація для scheduled mode.
4. Клієнт у калькуляторі бачить перемикач:
   - `Чим швидше`
   - `На запланований час`
5. Для термінового розрахунку використовується live GPS.
6. Для запланованого розрахунку використовується база техніки.
7. Доставка матеріалів рахує:
   - вартість матеріалу;
   - вартість доставки;
   - загальну суму.
8. Клієнт не бачить постачальника чи альтернативи.
9. Адмін бачить вибрану точку та альтернативні точки.
10. Заявка потрапляє в єдину CRM-вкладку `Заявки`.
11. З заявки можна створити `RentOrder` без нового окремого контуру.
12. Не зламано поточний евакуаторний калькулятор.
13. Не зламано execution flow, GPS, Telegram, BookedPeriod.
14. Не порушено архітектурні обмеження проєкту.

---

## 12. Що не робити

Не потрібно:

- створювати окрему публічну систему заявок поза поточним CRM;
- показувати клієнту постачальників;
- змушувати клієнта вибирати точку вручну;
- робити окрему вкладку заявок на матеріали;
- прив’язувати scheduled-розрахунок до live GPS;
- дублювати логіку калькулятора евакуатора і доставки, якщо її можна узгодити;
- робити великий косметичний редизайн замість функціональної реалізації.
