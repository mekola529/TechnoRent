# План реалізації виконання замовлення в декілька підходів / по змінах

## Ціль

Додати в TechnoRent сценарій, у якому одне замовлення може виконуватись:

- у кілька виїздів;
- у кілька змін;
- з паузами між етапами;
- без примусового фінального закриття після першого завершення працівником.

Рішення має спиратись на поточну архітектуру:

- `WorkAssignment`
- `WorkExecutionSession`
- `WorkExecutionReport`
- `OrderEventLog`
- Telegram worker flow

без радикальної заміни існуючого CRM-контуру.

---

## Поточний стан системи

Зараз логіка побудована як:

1. Працівник отримує `WorkAssignment`.
2. Для призначення використовується одна актуальна `WorkExecutionSession`:
   - старт;
   - завершення;
   - анкета;
   - фінальний звіт.
3. `RentOrder` переходить у `WORKER_COMPLETED`, коли:
   - усі прийняті призначення завершені;
   - у latest execution report стоїть `questionnaireStatus = COMPLETED`.

### Поточне обмеження

Система фактично трактує:

`одне призначення = один завершений цикл виконання`

Це погано підходить для кейсів:

- техніка виїхала сьогодні, але робота продовжиться завтра;
- працівник зробив лише частину задачі;
- об’єкт виконується по змінах;
- один і той самий працівник/техніка кілька разів повертаються на замовлення.

---

## Принцип майбутнього рішення

### Базова ідея

Не створювати нову верхньорівневу сутність замість `WorkExecutionSession`.

Потрібно використати вже наявну модель так:

- `WorkAssignment` = відповідальність конкретного працівника за конкретну техніку в межах замовлення;
- `WorkExecutionSession` = один окремий виїзд / одна зміна / один підхід;
- `WorkExecutionReport` = звіт по конкретній execution session;
- `WorkAssignment` закривається не після першої завершеної сесії, а тільки після фінального підтвердження, що всі підходи завершені.

Це найменш ризиковий шлях, бо не ламає основну модель даних.

---

## Яку поведінку має отримати користувач

### Для працівника в Telegram

Працівник має мати можливість:

1. Прийняти завдання.
2. Розпочати зміну / виїзд.
3. Завершити поточну зміну.
4. Відправити короткий звіт по зміні.
5. Вказати:
   - чи роботу виконано повністю;
   - чи потрібен ще один виїзд / ще одна зміна.

Якщо роботу не завершено повністю:

- assignment лишається активним;
- створюється завершена session за поточну зміну;
- працівник пізніше може запустити наступну session для того самого assignment.

Якщо роботу завершено повністю:

- assignment переходить у завершений стан;
- тільки тоді він може бути врахований як повністю виконаний для закриття замовлення.

### Для менеджера в CRM

Менеджер має бачити:

- не один execution status, а історію змін/виїздів;
- окремі сесії по кожному працівнику/техніці;
- проміжні звіти;
- фінальний статус assignment:
  - `очікує старту`
  - `в роботі`
  - `пауза / буде продовження`
  - `виконано повністю`

## Потрібні зміни в БД

## 1. Розширити `WorkAssignment`

Додати поля:

- `completionStatus` TEXT NOT NULL DEFAULT 'PENDING'`
- `completedAt` TIMESTAMPTZ NULL`
- `completionComment` TEXT NULL`
- `plannedNextStartAt` TIMESTAMPTZ NULL`

Рекомендовані значення `completionStatus`:

- `PENDING`
- `ACCEPTED`
- `IN_PROGRESS`
- `AWAITING_NEXT_SHIFT`
- `COMPLETED`
- `DECLINED`

### Навіщо

Зараз `status` assignment не покриває сценарій:

- працівник прийняв завдання;
- виконав частину;
- поїде ще раз;
- але assignment не повинен вважатись остаточно завершеним.

---

## 2. Розширити `WorkExecutionSession`

Додати поля:

- `sequenceNumber` INTEGER NOT NULL DEFAULT 1`
- `shiftLabel` TEXT NULL`
- `isFinalSession` BOOLEAN NOT NULL DEFAULT false`
- `sessionComment` TEXT NULL`

### Навіщо

Кожна session має бути окремою зміною / окремим підходом:

- `1 зміна`
- `2 зміна`
- `3 підхід`

`sequenceNumber` дасть стабільний порядок.

---

## 3. Розширити `WorkExecutionReport`

Додати session-level ознаки:

- `workCompleted` BOOLEAN NULL`
- `needsNextShift` BOOLEAN NULL`
- `nextShiftComment` TEXT NULL`

### Навіщо

Потрібно відрізняти:

- звичайне завершення зміни;
- фінальне завершення всієї роботи.

---

## 4. Прив’язати витрати і GPS-агрегацію до session

Усі автоматичні та ручні дані мають логічно прив’язуватись до конкретної зміни:

- GPS snapshot
- ручний пробіг
- мотогодини
- час руху
- час стоянки
- витрати працівника
- готівка від клієнта

`OrderExpense` можна лишити на рівні замовлення, але в payload або окремим nullable полем варто мати:

- `executionSessionId`

щоб менеджер розумів, до якої зміни належала витрата.

## Потрібні зміни в backend-логіці

## Етап 1. Змінити модель життєвого циклу assignment

Поточна логіка `readyToClose` дивиться на latest session.

Треба змінити її так:

- assignment вважається завершеним тільки якщо `completionStatus = COMPLETED`;
- finished latest session без фінального completion не дає права закривати замовлення.

### Конкретно

Потрібно переписати:

- `getOrderOperationalOverview(...)`
- `syncRentOrderOperationalStatus(...)`
- умови в `POST /api/admin/rent-orders/:id/close`

### Нове правило

Замовлення готове до фінального закриття лише коли:

1. усі релевантні assignment прийняті або коректно відхилені;
2. для кожного прийнятого assignment:
   - є хоча б одна execution session;
   - остання логічна session завершена;
   - assignment має `completionStatus = COMPLETED`.

---

## Етап 2. Дозволити багато session на одне assignment

Потрібно змінити flow:

### Старт виконання

Замість логіки:

- якщо session є, повторно використовувати її;

потрібно:

- якщо остання session `IN_PROGRESS` — не створювати нову;
- якщо остання session завершена і assignment ще не `COMPLETED` — створювати нову session з `sequenceNumber + 1`.

### Завершення виконання

Після `finish`:

- поточна session переходить у `FINISHED`;
- зберігається report по session;
- але assignment не стає `COMPLETED` автоматично.

---

## Етап 3. Розділити “звіт по зміні” і “фінальне завершення assignment”

Після завершення зміни Telegram має питати не лише про витрати / коментар, а й:

- `Роботу виконано повністю?`

Варіанти:

- `Так, завершено`
- `Ні, потрібна ще зміна`

### Якщо `Ні, потрібна ще зміна`

- session-report закривається;
- assignment переходить у `AWAITING_NEXT_SHIFT`;
- order лишається `ACTIVE`;
- працівник знову бачить завдання у списку;
- при наступному старті створюється нова session.

### Якщо `Так, завершено`

- session-report закривається;
- assignment переходить у `COMPLETED`;
- лише тоді він може враховуватись у `WORKER_COMPLETED`.

---

## Етап 4. GPS і пальне рахувати по всіх session

Зараз система сильно орієнтується на latest execution session.

Для multi-shift потрібно:

1. рахувати GPS-дані окремо по кожній session;
2. автоматичну витрату пального створювати по кожній session окремо;
3. у фінансах замовлення показувати:
   - витрати по session;
   - підсумок по assignment;
   - підсумок по order.

### Важливе правило

Ручні метрики повинні мати пріоритет над GPS не загалом по assignment, а по конкретній session.

---

## Етап 5. Оплата працівника для multi-shift

Є два нормальні сценарії:

1. Зарплата задається на весь assignment.
2. Зарплата накопичується по session.

### Рекомендований MVP

На першому етапі:

- не розбивати `WorkerCompensation` на окрему таблицю по session;
- зберегти компенсацію на рівні assignment;
- але для типів `hourly` і `shift` рахувати фактичну суму з агрегованих session-даних.

Приклад:

- `hourly` = сума годин по всіх session;
- `shift` = кількість завершених session;
- `manual/fixed` = без змін.

### Далі

Якщо знадобиться точніша фінансова модель, можна окремо вводити `WorkerSessionCompensation`, але це вже другий етап, не MVP.

---

## Етап 6. UI в адмінці

На сторінці замовлення потрібно додати новий блок:

- `Зміни / виїзди`

### У ньому показувати

По кожному assignment:

- працівник;
- техніка;
- статус assignment;
- список session зверху вниз.

Для кожної session:

- номер зміни;
- старт;
- завершення;
- джерело даних GPS;
- пробіг;
- мотогодини;
- витрати;
- готівка;
- коментар;
- фінальна ознака:
  - `потрібне продовження`
  - `роботу завершено`

### Окремо

У верхній частині worker-блоку має бути видно:

- `в роботі`
- `очікує наступну зміну`
- `завершено повністю`

без необхідності відкривати кожен лог.

## Етап 7. Telegram UX

Потрібно змінити worker flow:

### Меню “Мої завдання”

Показувати:

- активні assignment;
- окремо ті, що `AWAITING_NEXT_SHIFT`.

Кнопка має мінятись:

- `Розпочати першу зміну`
- `Розпочати наступну зміну`
- `Завершити поточну зміну`

### Після завершення зміни

Анкета має бути session-level:

- чи брав готівку;
- чи були витрати;
- чи були проблеми;
- коментар;
- чи це фінальне завершення роботи.

## Етап 8. Журнал подій

`OrderEventLog` має отримати нові event types:

- `worker_execution_session_started`
- `worker_execution_session_finished`
- `worker_execution_session_report_submitted`
- `worker_assignment_awaiting_next_shift`
- `worker_assignment_completed_final`

Це потрібно, щоб менеджер бачив не злиплий latest status, а всю історію виконання.

## Поетапний план реалізації

## Фаза 1. Підготовка даних

1. Додати нові поля в `WorkAssignment`.
2. Додати нові поля в `WorkExecutionSession`.
3. Додати нові поля в `WorkExecutionReport`.
4. Додати індекси по:
   - `WorkAssignment.completionStatus`
   - `WorkExecutionSession.assignmentId, sequenceNumber`

## Фаза 2. Backend state machine

1. Переписати `getOrderOperationalOverview`.
2. Переписати `syncRentOrderOperationalStatus`.
3. Оновити `close`-guard для замовлення.
4. Оновити create/start/finish logic для session.

## Фаза 3. Telegram flow

1. Додати сценарій `потрібна ще зміна`.
2. Додати повторний старт того самого assignment.
3. Розмежувати проміжний і фінальний звіт.

## Фаза 4. GPS / fuel / finance

1. Збирати GPS по всіх session.
2. Рахувати пальне по session.
3. Агрегувати session-дані в assignment summary.
4. Перевірити вплив на worker balance і order expenses.

## Фаза 5. CRM UI

1. Додати timeline змін.
2. Додати статуси `очікує наступну зміну` і `завершено повністю`.
3. Показати session-level витрати й GPS.

## Фаза 6. QA

Прогнати сценарії:

1. Один assignment, одна зміна.
2. Один assignment, дві зміни.
3. Два assignment, один завершений, інший очікує наступну зміну.
4. Ручні GPS-метрики в одній зі змін.
5. Часткова готівка та витрати по окремих змінах.
6. Фінальне закриття після останньої зміни.

---

## Основні ризики

1. Багато поточної логіки читає latest execution session як єдину істину.
2. Автоматичне пальне зараз розраховується з припущенням одного фінального execution context.
3. Worker questionnaire зараз є фактично фінальною дією, а не session-level звітом.
4. Фінальне закриття замовлення зараз занадто жорстко прив’язане до `completedReports === acceptedAssignments`.

## Критерії готовності

Функціонал вважається готовим, якщо:

1. Один worker може кілька разів запускати й завершувати зміну в межах одного assignment.
2. Замовлення не переходить у `WORKER_COMPLETED` після першої незавершеної зміни.
3. Менеджер бачить історію всіх session.
4. GPS і пальне сумуються по всіх session коректно.
5. Фінальний звіт не губиться між змінами.
6. Фінальне закриття доступне лише після реального завершення всіх assignment.
