# План реалізації вкладки «Налаштування сповіщень» в адмінці

## Поточний стан реалізації

Статус: **MVP завершено і протестовано локально 24.04.2026**.

- Реалізовано registry шаблонів сповіщень у backend-коді.
- Додано таблицю `NotificationTemplate` у `schema.ts`.
- Додано backend API для списку, деталей, редагування, reset, змінних і preview.
- Додано renderer із whitelist-перевіркою змінних та fallback на дефолтний шаблон.
- Додано сторінку `/admin/notifications` у поточний `AdminLayout`.
- Додано sidebar-пункт `Сповіщення`.
- Реалізовано UI: список шаблонів, фільтри, редактор, вставка змінних, preview, reset.
- Додано service-specific overrides: для кожного базового шаблону можна створити окремий текст під конкретну послугу.
- Базовий шаблон використовується як fallback, якщо для вибраної послуги немає окремого тексту.
- Підключено шаблони до ключових існуючих відправок: нові заявки адміну, service request адміну, manager dispatch, повідомлення працівнику при призначенні.
- Для нових заявок, призначення працівнику і manager dispatch подій backend автоматично передає `service.slug`, щоб обрати правильний override.
- Заповнено дефолтні тексти тих сповіщень, які вже використовувались у системі:
  - нова заявка з сайту адміну;
  - нова заявка на послугу адміну;
  - service-specific заявка евакуатора;
  - service-specific заявка доставки сипучих матеріалів;
  - service-specific заявка на вивіз будівельного сміття;
  - призначення працівнику;
  - service-specific призначення для евакуатора;
  - service-specific призначення для доставки матеріалів;
  - прийнято / відхилено / старт / фініш / звіт / фінальне закриття менеджеру.
- Фінальне закриття замовлення менеджером тепер також підключене до шаблону `order_closed_manager`.
- У повідомленні працівнику точки знову формуються як клікабельні Google Maps HTML-посилання, якщо є координати.
- Кнопки Telegram worker flow залишаються системними і не редагуються через UI.
- Перевірено: список шаблонів, service override, preview, reset service-specific дефолту, блокування невідомих змінних, `server build`, `client build`, `telegram-bot build`.

## Підсумок тестування

Локально перевірено через API:

- `GET /api/admin/notifications/templates` повертає 9 базових шаблонів.
- `GET /api/admin/notifications/templates/new_service_request_admin?serviceSlug=poslugy-evakuatora` повертає service-specific override.
- `GET /api/admin/notifications/templates/new_service_request_admin?serviceSlug=perevezennia-sypuchyh-materialiv` повертає service-specific override.
- `POST /api/admin/notifications/templates/new_service_request_admin/preview?serviceSlug=poslugy-evakuatora` рендерить блок `РОЗРАХУНОК`.
- `PUT` із невідомою змінною `{{unknown.value}}` повертає `400`.
- `POST /reset?serviceSlug=poslugy-evakuatora` відновлює дефолтний service-specific текст.

Збірки:

- `cd server && npm run build` — успішно.
- `cd client && npm run build` — успішно, лишається стандартне Vite-попередження про chunk size.
- `cd telegram-bot && npm run build` — успішно.

## Ціль

Додати в адмін-панель **окрему вкладку для керування сповіщеннями**, у якій менеджер/адмін зможе:

- бачити всі типи сповіщень, які вже існують у системі;
- вмикати / вимикати окремі сповіщення;
- редагувати шаблони текстів;
- вставляти змінні з вбудованого списку;
- бачити доступні змінні для конкретного типу сповіщення;
- переглядати preview з підставленими тестовими значеннями;
- керувати сповіщеннями **в межах уже існуючої CRM / Telegram / worker flow логіки**, без створення окремого паралельного контуру.

## Базовий принцип

Цю функцію потрібно реалізувати **не як окремий сервіс розсилок**, а як **конфігураційний шар над уже існуючими подіями системи**.

Тобто:

- події залишаються там, де вони зараз народжуються;
- бізнес-логіка заявок, замовлень, призначень, Telegram worker flow, execution flow і manager notifications не переноситься в новий модуль;
- новий модуль лише дає змогу **керувати шаблонами текстів і поведінкою сповіщень**;
- відправка надалі має використовувати поточні інтеграції (`server/lib/telegram.ts`, internal Telegram routes, worker bot flow тощо), але вже через шаблони з БД.

## Основний UX сторінки

Сторінка має складатися з 3 основних зон.

### 1. Ліва колонка — список типів сповіщень

Показати всі доступні шаблони у вигляді списку або таблиці.

Для кожного шаблону показувати:

- назву;
- внутрішній ключ;
- канал;
- статус: увімкнено / вимкнено;
- хто отримує;
- до якого сценарію належить;
- дата останнього редагування.

Фільтри зверху:

- канал (`telegram_admin`, `telegram_worker`, у майбутньому можна розширити);
- сценарій (`Заявки`, `Замовлення`, `Працівники`, `Execution`, `Системні`);
- статус (`всі`, `активні`, `вимкнені`);
- пошук по назві / ключу.

---

### 2. Центральна зона — редактор шаблону

При виборі конкретного типу відкривається редактор.

Поля редактора:

- `Назва шаблону`
- `Варіант для послуги`: базовий шаблон або конкретна послуга
- `Ключ`
- `Канал`
- `Сценарій`
- `Отримувач`
- `Активний` (toggle)
- `Текст шаблону`
- `Примітка для адміністратора` (необов’язково)

Для Telegram-шаблонів:

- окремо вказати, чи використовується HTML parse mode;
- попередити, що не можна ламати системні callback-кнопки;
- для шаблонів з кнопками редагується **тільки текстова частина повідомлення**, а не callback payload.

Кнопки дій:

- `Зберегти`
- `Скасувати`
- `Відновити дефолтний шаблон`
- `Попередній перегляд`

### Service-specific шаблони

Сповіщення може відрізнятися залежно від послуги. Наприклад, текст для евакуатора має містити дві точки маршруту, а текст для перевезення сипучих матеріалів — матеріал, кількість, точку постачання і адресу доставки.

Тому система працює так:

- кожен тип події має базовий шаблон;
- у редакторі можна вибрати конкретну послугу і зберегти окремий override;
- якщо override є, backend використовує його;
- якщо override немає, backend використовує базовий шаблон;
- reset для конкретної послуги повертає системний service-specific дефолт, якщо він є;
- якщо системного дефолту для конкретної послуги немає, reset видаляє override і повертає використання базового тексту;
- reset без вибраної послуги відновлює дефолтний базовий шаблон.

Технічно це реалізовано через поле `NotificationTemplate.serviceSlug`. Для стабільності використовуються окремі unique-індекси:

- один базовий шаблон на `key`, де `serviceSlug IS NULL`;
- один override на пару `key + serviceSlug`, де `serviceSlug IS NOT NULL`.

## Які сповіщення мають потрапити в систему

На першому етапі треба покрити лише ті сповіщення, які вже реально існують або прямо випливають із поточного CRM / Telegram flow.

### Група 1. Заявки

1. Нова заявка з сайту адміну
2. Нова заявка на послугу адміну
3. Нова CRM-заявка менеджеру (якщо окремо використовується)

### Група 2. Замовлення

4. Замовлення створено
5. Замовлення призначено працівнику
6. Замовлення змінено / оновлено
7. Замовлення фінально закрито менеджером

### Група 3. Worker / Telegram flow

8. Працівнику надіслано нове завдання
9. Працівник прийняв завдання
10. Працівник відхилив завдання
11. Працівник розпочав виконання
12. Працівник завершив виконання
13. Працівник завершив анкету після роботи
14. Менеджеру надійшов сигнал про готовий підсумковий звіт

### Група 4. Системні / сервісні

15. Повідомлення про помилку відправки (опціонально, якщо вже є сенс логічно підв’язати)
16. Сервісне повідомлення про недоступний шаблон / fallback (тільки внутрішньо, якщо потрібно)

> На першому етапі не потрібно вигадувати десятки нових подій. Треба спочатку покрити і централізувати вже існуючі повідомлення.

## Важливе правило щодо змінних

У кожного типу сповіщення має бути **власний whitelist змінних**.

Наприклад:

- для `worker_assignment_sent` можна використовувати `order.*`, `worker.*`, `equipment.*`, `gps.*`
- для `new_site_request_admin` — `request.*`, `service.*`, `system.*`
- для `execution_completed_manager` — `order.*`, `execution.*`, `report.*`, `worker.*`

Не можна робити один спільний необмежений список змінних для всіх шаблонів.

Причини:

- це плутає користувача;
- збільшує ризик помилок;
- ускладнює preview;
- робить систему менш контрольованою.

## Що саме редагується, а що ні

Потрібно чітко розділити:

### Редагується

- назва шаблону;
- увімкнено / вимкнено;
- текст повідомлення;
- примітка;
- можливо, parse mode / форматування.

### Не редагується через UI

- ключ події після створення;
- системна логіка, яка вирішує коли саме відправляти повідомлення;
- callback payload для inline-кнопок;
- внутрішні ID зв’язків;
- порядок виконання бізнес-процесу.

Це критично, щоб не зламати worker bot flow.

## Запропонована модель даних

Реалізувати через окремі таблиці, без PostgreSQL enum для бізнес-ключів.

### 1. `NotificationTemplate`

Призначення: основна таблиця шаблонів.

Поля:

- `id`
- `key` — унікальний текстовий ключ, наприклад `worker_assignment_sent`
- `serviceSlug` — nullable; `NULL` означає базовий шаблон, значення slug означає override для конкретної послуги
- `name`
- `channel` — текстове поле
- `category` — текстове поле
- `recipientType` — текстове поле
- `isEnabled` — boolean
- `bodyTemplate` — текст шаблону
- `notes` — nullable
- `isSystem` — boolean
- `createdAt`
- `updatedAt`
- `updatedByAdminId` — nullable

Унікальність:

- базовий шаблон: унікальний `key`, де `serviceSlug IS NULL`;
- service-specific override: унікальна пара `key + serviceSlug`, де `serviceSlug IS NOT NULL`.

### 2. `NotificationTemplateVariable`

Призначення: список змінних, дозволених для конкретного шаблону.

Поля:

- `id`
- `templateKey`
- `variableKey` — наприклад `order.id`
- `label`
- `description`
- `exampleValue`
- `sortOrder`

> Можна також не робити окрему таблицю для першої ітерації, а тримати registry у коді. Але з точки зору довгострокової зручності краще мати або окрему таблицю, або хоча б явний кодовий registry з API-видачею.

### 3. `NotificationTemplateVersion` (опціонально)

Не обов’язково для першої фази, але бажано закласти як наступний крок.

Дає змогу:

- бачити історію змін;
- відкотити шаблон;
- логувати хто і коли що змінив.

Для MVP можна не робити повну історію, але хоча б тримати `updatedAt` і `updatedByAdminId`.

## Реєстр типів сповіщень

Окрім БД потрібен **кодом описаний registry**, який буде джерелом істини для:

- ключів типів сповіщень;
- доступних змінних;
- дефолтних шаблонів;
- того, чи підтримує шаблон HTML;
- того, чи має повідомлення кнопки;
- який preview-builder використовувати.

Приклад структури:

- `key`
- `name`
- `channel`
- `category`
- `recipientType`
- `defaultTemplate`
- `supportsHtml`
- `hasInteractiveButtons`
- `allowedVariables[]`

Це дуже важливо, тому що не можна покладатися тільки на дані з БД без контрольованого registry.

## Notification service

Потрібно створити окремий сервіс, наприклад:

- `server/src/lib/notification-templates.ts`
- `server/src/lib/notification-renderer.ts`
- `server/src/lib/notification-service.ts`

### `notification-templates.ts`

Містить registry дефолтних шаблонів і опис дозволених змінних.

### `notification-renderer.ts`

Відповідає за:

- підстановку змінних;
- escape / HTML safety;
- перевірку, що всі змінні коректно підставились;
- формування preview.

### `notification-service.ts`

Відповідає за:

- отримання шаблону;
- fallback на дефолтний шаблон;
- рендер;
- виклик конкретного sender;
- централізовану точку відправки.

## API, які потрібно додати

### Admin API

#### 1. `GET /api/admin/notifications/templates`

Повертає список шаблонів з фільтрами.

#### 2. `GET /api/admin/notifications/templates/:key`

Повертає деталі конкретного шаблону.

#### 3. `PUT /api/admin/notifications/templates/:key`

Оновлює шаблон.

Редаговані поля:

- `name`
- `isEnabled`
- `bodyTemplate`
- `notes`

#### 4. `POST /api/admin/notifications/templates/:key/reset`

Скидає шаблон до дефолтного значення з registry.

#### 5. `GET /api/admin/notifications/templates/:key/variables`

Повертає список доступних змінних для шаблону.

#### 6. `POST /api/admin/notifications/templates/:key/preview`

Повертає preview з тестовими даними.

Опціонально:

#### 7. `POST /api/admin/notifications/templates/:key/test-send`

Тільки якщо є безпечний і зрозумілий сценарій тестової відправки.

Для першої ітерації можна не робити.

## Важливі технічні правила

### 1. Fallback обов’язковий

Якщо шаблон не знайдений у БД або випадково зламаний:

- система не повинна падати;
- має використовуватись дефолтний шаблон із registry.

### 2. Валідація змінних

При збереженні шаблону:

- знайти всі `{{...}}` токени;
- перевірити, що вони входять у whitelist шаблону;
- якщо є невідома змінна — не зберігати шаблон і показати помилку.

### 3. Не ламати кнопки Telegram

Для шаблонів, де є inline-кнопки:

- редагується тільки текст;
- кнопки і callback data контролюються кодом.

### 4. Не використовувати Postgres enum для нових бізнес-ключів

Ключі каналів, категорій, recipientType та template keys краще тримати як `TEXT` + backend validation.

### 5. Без секретів у frontend

Ніякі токени, chat id чи інші секрети не повинні потрапити в client-side bundle.

## Поетапний план реалізації

# Етап 1. Проєктування і registry

### Завдання

- визначити повний список notification types для першої ітерації;
- створити registry у коді;
- зафіксувати allowed variables для кожного типу;
- визначити, які шаблони мають HTML;
- визначити, які шаблони мають кнопки і повинні залишатися частково системними.

### Результат

- один централізований registry-файл;
- чіткий список типів сповіщень;
- зрозумілий contract для БД і API.

---

# Етап 2. База даних і backend API

### Завдання

- додати таблицю `NotificationTemplate`;
- додати, за потреби, таблицю або registry-видачу для змінних;
- оновити `schema.ts`;
- додати CRUD/read API для сторінки адмінки;
- додати preview endpoint;
- додати reset endpoint.

### Результат

- шаблони зберігаються в БД;
- є API для списку, деталей, редагування, reset і preview.

---

# Етап 3. Notification renderer і fallback логіка

### Завдання

- реалізувати рендер змінних;
- реалізувати whitelist validation;
- реалізувати fallback на default template;
- зробити sample data builders для preview;
- підготувати один централізований `sendNotification()`.

### Результат

- є надійний механізм рендеру;
- шаблони не ламають систему;
- preview відповідає реальному формату.

---

# Етап 4. Фронтенд адмінки

### Завдання

- додати новий route `/admin/notifications`;
- додати пункт у sidebar;
- реалізувати сторінку списку шаблонів;
- реалізувати редактор шаблону;
- реалізувати праву панель зі змінними;
- реалізувати preview;
- реалізувати dirty-state, валідацію, success/error state.

### Результат

- повноцінна керована UI-сторінка для редагування сповіщень.

---

# Етап 5. Інтеграція з існуючими подіями

### Завдання

Поступово перевести вже існуючі відправки повідомлень на новий механізм.

Пріоритет:

1. Нові заявки адміну
2. Призначення працівнику
3. Прийнято / відхилено
4. Старт / завершення виконання
5. Готовий підсумковий звіт менеджеру
6. Фінальне закриття замовлення

### Результат

- фактичні повідомлення реально використовують шаблони з адмінки;
- зберігається поточна бізнес-логіка.

### Перевірити

- збереження шаблонів;
- reset до дефолту;

- валідацію змінних;
- fallback при битому шаблоні;
- worker flow після інтеграції;
- manager notifications;

- `npx tsc --noEmit`;

### Результат

- існуючий Telegram flow не зламаний.

## Пріоритет MVP

Для першої робочої версії достатньо:

- нової вкладки `Налаштування сповіщень`;
- списку шаблонів;
- редагування тексту;
- списку змінних;
- preview;
- reset to default;
- інтеграції з 3–5 найважливішими типами повідомлень.

Не обов’язково одразу робити:

- історію версій;
- тестову реальну відправку;
- багатоканальність beyond Telegram;
- drag-and-drop конструктор.

## Критерії успіху

Результат вважається добрим, якщо:

- у sidebar з’явилась окрема вкладка налаштувань сповіщень;
- всі ключові типи повідомлень видно в одному місці;
- шаблони можна редагувати без втручання в код;
- змінні видно збоку і можна вставляти кліком;
- preview працює коректно;
- некоректні змінні блокують збереження;
- fallback не дає системі зламатися;
- поточний CRM / Telegram / worker flow лишається робочим.

---

## Що не потрібно робити зараз

- не створювати окремий мікросервіс сповіщень;
- не робити окрему панель поза поточною адмінкою;
- не додавати складну систему ролей саме для цієї сторінки, якщо це ще не зроблено глобально;
- не дозволяти редагувати callback payload Telegram-кнопок;
- не намагатися одразу покрити всі можливі майбутні канали;
- не ламати існуючі місця відправки без fallback.
