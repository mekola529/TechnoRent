
# Фінальна комплексна перевірка CRM, фінансів, GPS, UI/UX, документації та підготовка до деплою

Ти вже в контексті проєкту TechnoRent. Потрібно провести фінальну комплексну перевірку всього реалізованого функціоналу CRM, фінансового модуля, GPS-логіки, Telegram-флоу, UI/UX, документації та підготувати проєкт до деплою.

Також потрібно окремо перевірити й покращити UX/UI фінансового блоку в замовленнях CRM на сторінці `/admin/rent-orders`, бо зараз він виглядає перевантажено: багато великих блоків і форм одночасно видно на екрані, сторінка дуже довга, менеджеру складно швидко зрозуміти головне: яка сума замовлення, скільки клієнт оплатив, який борг, скільки потрібно заплатити працівнику, чи є витрати, чи можна закривати замовлення.

Головна задача — не переписати все з нуля, а стабілізувати, перевірити, спростити UI, виправити помилки, оновити документацію і підготувати проєкт до production-деплою.

---

## 0. Важливі правила

Працюй дуже обережно.

Не ламай існуючу бізнес-логіку.

Не видаляй функціонал без причини.

Не видаляй уже створені API, таблиці, поля та розрахунки.

Не змінюй назви полів без крайньої потреби.

Не ламай створення/редагування замовлення.

Не ламай створення замовлення із заявки.

Не ламай призначення працівника.

Не ламай Telegram-флоу працівника.

Не ламай GPS cron і загальну синхронізацію для мапи.

Не ламай фінальне закриття менеджером.

Не ламай журнал подій.

Не змінюй backend, якщо задачу можна вирішити тільки на frontend.

Якщо знаходиш баг — спочатку зрозумій причину, потім виправ.

Після кожного суттєвого виправлення перевір, що пов’язаний функціонал не зламався.

Не додавай зайвих великих фіч. Потрібні тільки перевірка, стабілізація, UI/UX-покращення, виправлення, документація і підготовка до деплою.

Після змін обов’язково перевір, що build проходить без помилок.

## 2. Окремий обов’язковий блок: покращення UI/UX фінансів у замовленні

### 2.1. Поточна проблема

На сторінці `/admin/rent-orders` фінансова частина зараз перевантажена:

- багато блоків і форм видно одночасно;
- сторінка дуже довга;
- багато полів займають місце навіть коли вони не потрібні;
- менеджеру важко швидко зрозуміти фінансовий стан замовлення;
- форми додавання позицій, оплат, витрат і розрахунків відкриті постійно або займають забагато простору.

Потрібно зробити фінансову частину компактнішою і зрозумілішою.

### 2.2. Основна ідея редизайну

Замість того, щоб показувати всі фінансові форми одразу, потрібно зробити:

1. Короткий фінансовий підсумок зверху.
2. Далі фінансові блоки у вигляді accordion-секцій.
3. Форми додавання позицій, оплат, витрат і розрахунків не мають бути відкриті постійно.
4. Кожна форма відкривається тільки після натискання кнопки `+ Додати`.
5. Після збереження або скасування форма згортається.
6. Менеджер має бачити головне одразу, а деталі відкривати тільки коли потрібно.

### 2.3. Бажана структура фінансової частини замовлення

У картці замовлення фінансову частину потрібно організувати так:

1. `Фінанси замовлення` — короткий підсумок у вигляді KPI.
2. Accordion-секція `Розрахунок для клієнта`.
3. Accordion-секція `Оплати клієнта`.
4. Accordion-секція `Витрати замовлення`.
5. Accordion-секція `Оплата працівника`.
6. Accordion-секція `Розрахунок з працівником`.
7. Accordion-секція `Фінальне закриття`.
8. Accordion-секція `Журнал подій`.

За замовчуванням:

- фінансовий підсумок завжди видимий;
- `Розрахунок для клієнта` можна залишити відкритим;
- інші секції зробити згорнутими;
- `Журнал подій` також згорнутий за замовчуванням.

### 2.4. Верхній блок “Фінанси замовлення”

Перероби поточний великий блок фінансів у компактний підсумок.

Показати KPI-картки:

- Розраховано;
- Погоджено з клієнтом;
- Оплачено;
- Борг клієнта;
- Витрати замовлення;
- Оплата працівника;
- Баланс з працівником;
- Прибуток.

Логіка відображення:

- якщо борг клієнта > 0 — візуально виділити;
- якщо клієнт оплатив повністю — показати позитивний статус;
- якщо прибуток від’ємний — виділити як проблему;
- якщо розрахунок з працівником не закритий — показати попередження.

Поля `Погоджена сума` і `Фінансовий коментар` не повинні постійно займати багато місця.

Зроби кнопку:

`Редагувати фінансові дані`

Після натискання відкривай компактну inline-форму або невеликий блок редагування.

Після збереження блок має згортатися назад.

Не змінюй існуючу логіку збереження погодженої суми, просто покращ її відображення.

### 2.5. Accordion-компонент

Створи або використай універсальний компонент для згортання секцій, наприклад `AdminAccordionSection`.

Він має підтримувати:

- title;
- subtitle;
- badge або короткий статус;
- defaultOpen;
- children.

Візуально accordion має відповідати стилю адмінки:

- світла картка;
- border;
- rounded corners;
- акуратний заголовок;
- іконка/індикатор відкриття;
- без різких змін глобального дизайну.

### 2.6. Секція “Розрахунок для клієнта”

Зараз форма позицій розрахунку займає багато місця. Потрібно зробити її компактною.

Спочатку показуй таблицю позицій:

| Назва | Кількість | Ціна | Сума | Дії |
|---|---:|---:|---:|---|

Якщо позицій немає, показати короткий empty state:

`Позицій ще немає.`

І кнопку:

`+ Додати позицію`

Форма додавання позиції не має бути постійно відкритою. Вона відкривається тільки після натискання `+ Додати позицію`.

У формі мають бути поля:

- шаблон;
- назва;
- тип розрахунку;
- кількість;
- одиниця;
- ціна / сума;
- техніка;
- коментар.

Форму зробити компактною:

- desktop: 2–3 колонки залежно від ширини;
- mobile: 1 колонка;
- кнопки внизу: `Зберегти позицію` і `Скасувати`.

Якщо обрано шаблон, автоматично заповнювати:

- назву;
- тип;
- одиницю;
- ціну.

Типи розрахунку мають працювати логічно:

- фіксована сума — кількість за замовчуванням 1;
- за км — кількість означає кілометри;
- за годину — кількість означає години;
- ручна сума — дозволити ввести фінальну суму вручну.

Після збереження позиції форма має згортатися.

### 2.7. Секція “Оплати клієнта”

Зроби блок оплат компактним.

Спочатку показати короткий підсумок:

- Оплачено клієнтом;
- Борг клієнта;
- Статус оплати.

Потім кнопку:

`+ Додати оплату`

Нижче таблиця оплат:

| Дата | Сума | Метод | Хто отримав | Коментар | Дії |
|---|---:|---|---|---|---|

Форма додавання оплати відкривається тільки після натискання `+ Додати оплату`.

Поля форми:

- сума;
- метод оплати;
- хто отримав: менеджер / працівник;
- працівник, якщо отримав працівник;
- дата і час;
- коментар.

Після додавання оплати:

- оновити фінансовий підсумок;
- оновити борг клієнта;
- оновити статус оплати;
- форма має згортатися.

### 2.8. Секція “Витрати замовлення”

Зроби блок витрат компактним.

Спочатку показати:

- загальна сума витрат замовлення;
- кнопку `+ Додати витрату`.

Таблиця витрат:

| Дата | Тип | Сума | Техніка | Працівник | Джерело | Коментар | Дії |
|---|---|---:|---|---|---|---|---|

Форма додавання витрати відкривається тільки після натискання `+ Додати витрату`.

Поля форми:

- тип витрати;
- сума;
- джерело: менеджер / працівник;
- працівник;
- техніка;
- дата і час;
- коментар.

Якщо витрата прийшла зі звіту працівника, вона має бути зрозуміло позначена як `Джерело: працівник`.

Не змішуй її в UI так, щоб було незрозуміло, хто додав витрату.

Після збереження форма має згортатися.

### 2.9. Секція “Оплата працівника”

Поточну форму оплати працівника потрібно зробити динамічною.

Не показуй всі поля одночасно.

Завжди показувати:

- працівник;
- тип оплати;
- коментар.

Далі поля мають залежати від типу оплати.

#### Якщо тип “Фіксовано”

Показати тільки:

- сума.

#### Якщо тип “За годину”

Показати:

- ставка за годину;
- кількість годин;
- фінальна сума автоматично.

#### Якщо тип “За зміну”

Показати:

- ставка за зміну;
- кількість змін;
- фінальна сума автоматично.

#### Якщо тип “Відсоток”

Показати:

- відсоток;
- база розрахунку;
- орієнтовна сума автоматично.

#### Якщо тип “Ручна сума”

Показати:

- фінальна сума.

Не показуй поля `ставка`, `кількість`, `відсоток`, `фінальна сума` всі одночасно, якщо вони не потрібні для вибраного типу.

Це головне спрощення цього блоку.

Перевір, що payload на backend лишається сумісним з уже реалізованою логікою.

### 2.10. Секція “Розрахунок з працівником”

Зроби цей блок як підсумок + таблиця + кнопка.

Показати коротко:

- Працівник має отримати;
- Працівник отримав від клієнта;
- Витрати працівника;
- Баланс;
- Статус розрахунку.

Приклад:

`Баланс: працівник має передати компанії 300 грн`

або:

`Баланс: компанія винна працівнику 1500 грн`

Потім кнопка:

`+ Додати розрахунок`

Таблиця розрахунків:

| Дата | Напрямок | Сума | Метод | Коментар | Дії |
|---|---|---:|---|---|---|

Форма додавання розрахунку відкривається тільки після натискання кнопки.

Поля форми:

- працівник;
- сума;
- напрямок;
- метод;
- дата;
- коментар.

Після збереження форма має згортатися.

### 2.11. Секція “Фінальне закриття”

Цей блок не має плутати користувача.

Якщо фінальне закриття ще недоступне, показати тільки коротке пояснення:

`Фінальне закриття стане доступним після завершення роботи працівником і заповнення підсумкового звіту.`

Не показуй велику форму, якщо закриття ще недоступне.

Якщо закриття доступне:

- показати фінальний фінансовий підсумок;
- показати попередження, якщо клієнт має борг;
- показати попередження, якщо розрахунок з працівником не закритий;
- залишити кнопку `Закрити замовлення`.

Не змінюй умови доступності фінального закриття без потреби.

### 2.12. Секція “Журнал подій”

Журнал подій залишити внизу, але зробити згорнутим за замовчуванням.

У заголовку показати кількість подій:

`Журнал подій (5)`

Розгортання по кліку.

Сам журнал подій не ламати. Якщо є існуюча логіка завантаження або відображення — зберегти її.

### 2.13. Загальні UI-вимоги для редизайну

Зберігай стиль адмінки:

- світлі картки;
- округлення;
- жовтий primary;
- темний sidebar;
- акуратні borders;
- існуючі admin-компоненти, якщо вони є.

Покращення мають бути саме в UX:

- менше одночасно відкритих форм;
- більше коротких підсумків;
- більше таблиць;
- більше кнопок `+ Додати`;
- більше згортання/розгортання;
- менше великих порожніх полів;
- не робити сторінку ще довшою.

### 2.14. Mobile-вимоги для редизайну

Обов’язково перевірити мобільну версію:

- 375px;
- 390px;
- 430px.

На мобільній:

- усі форми в одну колонку;
- кнопки мають бути зручними для натискання;
- не має бути загального горизонтального scroll сторінки;
- якщо таблиці не влазять, зробити горизонтальний scroll тільки всередині таблиці або адаптувати записи в картки;
- accordion має нормально відкриватися й закриватися.

### 2.15. Етапи виконання UI/UX редизайну

#### Етап UI-1. Аналіз

1. Відкрити сторінку `AdminRentOrdersPage`.
2. Знайти блоки:
   - фінанси замовлення;
   - позиції розрахунку;
   - оплати клієнта;
   - витрати замовлення;
   - оплата працівника;
   - розрахунок з працівником;
   - фінальне закриття;
   - журнал подій.
3. Зрозуміти, які state, API-запити і типи вже використовуються.
4. На цьому етапі не змінювати backend.

#### Етап UI-2. Фінансовий підсумок

1. Переробити блок `Фінанси замовлення` у компактну KPI-сітку.
2. Погоджену суму і фінансовий коментар винести в режим редагування.
3. Додати кнопку `Редагувати фінансові дані`.
4. Перевірити, що збереження працює як раніше.

#### Етап UI-3. Accordion-структура

1. Створити або використати accordion-компонент.
2. Обгорнути фінансові блоки в accordion-секції.
3. За замовчуванням відкрити тільки потрібні секції.
4. Журнал подій зробити згорнутим.

#### Етап UI-4. Приховані форми додавання

1. Прибрати постійно відкриті форми.
2. Додати кнопки:
   - `+ Додати позицію`;
   - `+ Додати оплату`;
   - `+ Додати витрату`;
   - `+ Додати розрахунок`.
3. Форми відкривати тільки після кліку.
4. Після збереження або скасування форма має згортатися.

#### Етап UI-5. Динамічна оплата працівника

1. Залежно від типу оплати показувати тільки потрібні поля.
2. Не показувати всі поля одночасно.
3. Перевірити, що фінальна сума рахується правильно.
4. Перевірити, що backend отримує сумісний payload.

#### Етап UI-6. Таблиці та empty states

1. Якщо записів немає — показувати короткий empty state.
2. Якщо записи є — показувати компактну таблицю.
3. Додаткові коментарі не мають розтягувати таблиці.
4. На мобільній версії таблиці мають бути адаптивними.

#### Етап UI-7. Фінальне закриття

1. Якщо недоступне — показувати тільки коротке пояснення.
2. Якщо доступне — показувати підсумок, попередження і кнопку.
3. Не змінювати бізнес-умови закриття без потреби.

## 3. Підготовка тестових даних

Створи або використай тестові дані, щоб перевірити повний сценарій.

### 3.1. Тестова техніка

Переконайся, що є мінімум:

## 4. Перевірка повного сценарію заявки → замовлення → виконання → фінанси

### 4.1. Сценарій 1. Заявка з публічного сайту

Перевір:

2. Заявка має з’явитися у вкладці `Заявки`.
3. Дані клієнта мають бути коректними:
   - ім’я;
   - телефон;
   - послуга/техніка;
   - адреса;
   - коментар;
   - попередній розрахунок, якщо він є.
4. Якщо заявка створена з калькулятора, перевір, що приблизний розрахунок зберігається і видимий менеджеру.

### 4.2. Сценарій 2. Створення замовлення з заявки

Перевір:

1. Менеджер відкриває заявку.
2. Створює з неї замовлення.
3. Дані заявки коректно переносяться у замовлення.
4. Якщо є попередній розрахунок — перевір можливість підтягнути його в позиції розрахунку.
5. Замовлення з’являється у вкладці `Замовлення`.

### 4.3. Сценарій 3. Позиції розрахунку

Перевір:

1. Додати позицію з шаблону.
2. Додати ручну позицію.
3. Додати позицію за км.
4. Додати позицію за годину.
5. Перевір, що сума позиції рахується правильно:
   - кількість × ціна;
   - ручна сума;
   - фіксована сума.
6. Перевір, що загальна розрахована сума оновлюється.
7. Перевір, що погоджену суму можна зберегти окремо.
8. Перевір, що фінансовий підсумок оновлюється після змін.

### 4.4. Сценарій 4. Оплати клієнта

Перевір:

1. Додати оплату готівкою.
2. Додати оплату на карту/рахунок, якщо такі методи є.
3. Додати оплату, яку отримав працівник.
4. Перевір:
   - оплачено;
   - борг клієнта;
   - статус оплати.
5. Статуси мають працювати:
   - не оплачено;
   - частково оплачено;
   - оплачено;
   - переплата, якщо реалізовано.

### 4.5. Сценарій 5. Витрати замовлення

Перевір:

1. Додати витрату замовлення від менеджера.
2. Додати витрату, прив’язану до працівника.
3. Додати витрату, прив’язану до техніки.
4. Перевір, що витрати входять у фінансовий підсумок замовлення.
5. Перевір, що прибуток перераховується правильно.

### 4.6. Сценарій 6. Оплата працівника

Перевір усі типи оплати:

1. Фіксовано.

3. За зміну.

### 4.7. Сценарій 7. Призначення працівника

6. У журналі подій з’являється відповідний запис.

## 6. Перевірка GPS-логіки

Особливо перевір нову GPS-логіку.

### 6.1. Що має працювати

1. При старті виконання:
   - береться актуальний snapshot напряму з `gps.equgps.com`;
   - не використовувати локальну БД як основне джерело для snapshot виконання;
   - зберегти стартові GPS-дані.

2. При завершенні:
   - знову зробити прямий запит до `gps.equgps.com`;
   - зберегти фінальні GPS-дані.

3. Після завершення розрахувати:
   - кілометраж;
   - час у русі;
   - простій;
   - мотогодини.

4. Якщо EquGPS підтримує звіт за інтервал:
   - використовувати інтервал `startedAt → finishedAt`;
   - це має бути основним способом розрахунку.

5. Якщо інтервальний звіт не спрацював:
   - fallback на різницю між стартовим і фінальним snapshot;
   - не блокувати завершення роботи;
   - показати менеджеру попередження.

6. У CRM має бути видно:
   - стартову адресу;
   - фінальну адресу;
   - стартові координати;
   - фінальні координати;
   - кілометраж;
   - час руху;
   - простій;
   - мотогодини;
   - статус GPS-даних;
   - помилку, якщо вона була.

### 6.2. Важливо

Не ламай існуючий GPS cron і загальну синхронізацію для мапи.

Поточний sync у локальну БД має залишитися для:

- вкладки `Мапа`;
- поточних позицій;
- денної статистики;
- поїздок;
- стоянок.

Але для GPS-звіту конкретного виконання замовлення основним джерелом має бути прямий запит до `gps.equgps.com`.

## 7. Перевірка фінального закриття замовлення

Перевір:

1. Якщо працівник ще не завершив роботу — фінальне закриття недоступне.
2. Якщо працівник завершив, але не заповнив звіт — фінальне закриття недоступне.
3. Якщо звіт заповнений — менеджер може закрити замовлення.

   - суму замовлення;

   - борг клієнта;
   - витрати;
   - зарплату працівника;
   - баланс з працівником;
   - прибуток.
5. Якщо є борг клієнта — показати попередження.
6. Якщо розрахунок з працівником не закритий — показати попередження.
7. Після фінального закриття:
   - статус замовлення має змінитися на завершений;
   - фінансові дані не мають зникнути;
   - замовлення має враховуватися у фінансовій аналітиці.

- `telegram-bot/dist/`
- `client/dist/` або потрібний `client_dist/`
- `server/package.json`
- `telegram-bot/package.json`
- production dependencies;
- `.env.example`;
- docs.

### 16.2. Не включати в архів

Не включати:

- `.env` з реальними секретами;
- локальні логи, якщо вони не потрібні;
- кеші;
- `.DS_Store`;
- зайві локальні файли.

Щодо `node_modules`:

- якщо поточна cPanel-інструкція проєкту передбачає завантаження `node_modules`, дотримуватись саме поточної документації проєкту;
- якщо інструкція передбачає встановлення залежностей на сервері — не включати `node_modules`.

### 16.3. Перевірити env

Перевір, що в `.env.example` є всі потрібні змінні:

- `DATABASE_URL`
- `JWT_SECRET`
- `CLIENT_URL`
- `SITE_URL`
- `TELEGRAM_BOT_TOKEN`
- `TELEGRAM_CHAT_ID`
- `TELEGRAM_INTERNAL_TOKEN`
- `TELEGRAM_BOT_INTERNAL_URL`
- `TELEGRAM_BOT_PORT`
- `TELEGRAM_BOT_WEBHOOK_SECRET`
- `BACKEND_INTERNAL_URL`
- `EQUGPS_PLATFORM_URL`
- `EQUGPS_EMAIL`
- `EQUGPS_PASSWORD`

Якщо для фінансів або Excel додались нові env-змінні — додати їх у `.env.example` і документацію.

В кінці роботи підготуй короткий звіт.

### 17.1. Що перевірено

Перелічити:

- заявки;
- замовлення;
- фінанси замовлення;
- позиції розрахунку;
- оплати;
- витрати;
- оплата працівника;
- розрахунок з працівником;
- працівники;
- Telegram;
- GPS;

- витрати техніки;
- Excel;
- UI/UX;
- mobile;
- build;
- документація;
- деплойна структура.

### 17.2. Що виправлено

Перелічити конкретно:

- файл;
- проблема;
- що змінено.

### 17.3. Що залишилось на потім

Якщо щось не реалізовано або потребує окремого етапу, вказати:

- фото чеків;

- розмежування ADMIN/MANAGER;
- облік залишків пального;
- додаткова аналітика;
- інше, якщо знайдено.

Наприклад:

```bash
cd server && npm run build
cd client && npm run build
cd telegram-bot && npm run build
```

### 17.5. Результат

Написати чітко:

- чи готово до деплою;
- чи є критичні проблеми;
- чи потрібно щось вручну перевірити перед production.

## 18. Очікуваний фінальний результат

Після виконання цього завдання проєкт має бути готовий до деплою:

- CRM працює стабільно;
- фінансовий модуль рахує коректно;
- UI фінансів у замовленні став простішим і зручнішим;
- GPS-дані для виконання беруться правильно;
- Telegram-флоу не зависає;
- UI/UX зручний;
- мобільна версія не ламається;
- Excel-експорт працює;
- документація актуальна;
- build проходить;
- структура для production-деплою готова.
