# Промпт для ШІ агента: сторінка послуг + зв’язок послуг і техніки в обидві сторони

## Загальна ціль
Потрібно реалізувати на сайті повноцінний розділ **послуг** і пов’язати його з уже існуючим розділом **техніки** так, щоб користувач міг:

1. перейти на сторінку **всіх послуг**;
2. переглянути картки послуг;
3. відкрити **окрему сторінку конкретної послуги**;
4. на сторінці послуги побачити **яка техніка може виконувати цю роботу**;
5. на сторінці конкретної техніки побачити **які види робіт може виконувати ця техніка**;
6. перейти зі сторінки техніки на сторінку відповідної послуги через кнопку **«Детальніше»**.

Рішення має бути цілісним, масштабованим і працювати в обидві сторони:
- **послуга → техніка**;
- **техніка → послуга**.

---

## Основна задача
### 1. Додати окрему сторінку послуг
На сторінці послуг мають бути представлені різні види робіт у вигляді карток.

Кожна картка повинна містити:
- зображення;
- назву послуги;
- короткий опис;
- клік по картці або кнопка для переходу на окрему сторінку послуги.

### 2. Створити окремі сторінки кожної послуги
На сторінці послуги потрібно показати:
- назву послуги;
- тематичне зображення або hero-блок;
- детальний опис;
- приблизну вартість або логіку розрахунку;
- список техніки, яка може виконувати цю послугу;
- CTA-блок із заявкою або консультацією.

### 3. Оновити сторінку кожної техніки
На сторінці конкретної техніки потрібно додати новий блок:

## «Види робіт, які може виконувати ця техніка»

У цьому блоці мають відображатися пов’язані послуги.

Для кожної послуги в блоці потрібно показати:
- назву роботи / послуги;
- за бажанням короткий підпис або 1 короткий рядок опису;
- кнопку **«Детальніше»**, яка веде на сторінку відповідної послуги.

Це означає, що сторінка техніки повинна мати прямі переходи на відповідні сторінки послуг.

---

## Приклади послуг, які потрібно додати
Обов’язково додати:
1. Вивіз будівельного сміття
2. Послуги евакуатора
3. Копання траншеї

Також додати ще кілька послуг для прикладу, але тільки тих, які реально можна виконати технікою з каталогу:
4. Планування ділянки
5. Риття котлованів
6. Демонтажні роботи
7. Завантаження та вивезення ґрунту
8. Перевезення сипучих матеріалів
9. Очищення території від будівельного сміття
10. Доставка будівельної техніки на об’єкт
11. Підготовка ділянки до будівництва
12. Земляні роботи на приватних і комерційних об’єктах

Можна додати ще 2–4 релевантні послуги, але **тільки якщо вони відповідають техніці, яка реально є на сайті**.

---

## Критично важливе правило
**Не можна вигадувати техніку, якої немає на сайті.**

Потрібно використовувати тільки ту техніку, яка вже є в каталозі / базі / JSON / API сайту.

Якщо для певної послуги немає відповідної техніки на сайті:
- або не додавати цю послугу;
- або позначити її як таку, що поки не повинна відображатися;
- або реалізувати гнучку систему фільтрації, яка покаже лише валідні послуги.

---

# Етапи виконання

## Етап 1. Проаналізувати поточну структуру проєкту
1. Проаналізуй структуру проєкту.
2. Знайди, де зберігаються дані про техніку:
   - JSON;
   - масиви;
   - API;
   - база даних;
   - mock-дані.
3. Визнач, які одиниці техніки реально є на сайті.
4. Знайди, як побудовані:
   - сторінка списку техніки;
   - сторінка окремої техніки;
   - маршрутизація;
   - компоненти карток.
5. Визнач, які послуги можна пов’язати з уже існуючою технікою.

---

## Етап 2. Спроєктувати структуру даних послуг
Створи окрему структуру даних для послуг.

Для кожної послуги бажано передбачити:
- `slug`
- `title`
- `shortDescription`
- `fullDescription`
- `image`
- `priceInfo`
- `pricingType`
- `relatedEquipment`
- `features`
- `seoTitle`
- `seoDescription`

### Приклад структури
```ts
 type Service = {
  slug: string;
  title: string;
  shortDescription: string;
  fullDescription: string;
  image: string;
  priceInfo: string;
  pricingType: "fixed_from" | "hourly_from" | "calculator" | "custom";
  relatedEquipment: string[];
  features: string[];
  seoTitle: string;
  seoDescription: string;
};
```

`relatedEquipment` має містити `id`, `slug` або інші унікальні ключі техніки, яка вже існує на сайті.

---

## Етап 3. Спроєктувати двосторонній зв’язок між послугами і технікою
Потрібно реалізувати таку логіку, щоб зв’язок працював в обидві сторони.

### Варіант реалізації
- Джерелом істини для зв’язків може бути масив послуг, де кожна послуга має `relatedEquipment`.
- На сторінці послуги техніка підтягується через `relatedEquipment`.
- На сторінці техніки список доступних робіт / послуг визначається шляхом фільтрації всіх послуг, у яких `relatedEquipment` містить `id` або `slug` поточної техніки.

### Важливо
1. Не дублювати зв’язки в двох місцях без потреби.
2. Краще зробити **одне джерело істини**.
3. Зв’язки мають бути стійкими до змін у даних.
4. Якщо послуга не має валідної техніки — вона не повинна ламати UI.
5. Якщо техніка не має жодної пов’язаної послуги — показувати блок лише за наявності даних або fallback-повідомлення.

---

## Етап 4. Створити сторінку списку послуг
Створи окрему сторінку, наприклад:
- `/services`
- або `/poslugy`

На сторінці потрібно:
1. Додати заголовок і короткий вступ.
2. Вивести послуги у вигляді адаптивної сітки карток.
3. Для кожної картки показати:
   - зображення;
   - назву;
   - короткий опис;
   - CTA «Детальніше».
4. Зробити картки клікабельними.
5. Додати охайні hover-ефекти.
6. Забезпечити гарний вигляд на мобільних, планшетах і десктопі.

---

## Етап 5. Створити сторінки окремих послуг
Для кожної послуги створи окрему сторінку, наприклад:
- `/services/vyviz-budivelnogo-smittya`
- `/services/evakuator`
- `/services/kopannya-transheyi`

На сторінці послуги повинні бути такі блоки:

### 1. Hero-блок
- назва послуги;
- короткий підзаголовок;
- тематичне зображення;
- CTA-кнопка.

### 2. Детальний опис
- що входить у послугу;
- для кого вона підходить;
- коли її замовляють;
- переваги.

### 3. Блок вартості
Потрібно реалізувати один із форматів:
- «від ... грн»;
- «від ... грн/год»;
- приблизний розрахунок;
- індивідуальний розрахунок.

Не вигадуй нереалістичні ціни.
Якщо точних цін немає, краще використовуй:
- «вартість залежить від обсягу робіт»;
- «ціна розраховується індивідуально»;
- «від ... грн» лише якщо така логіка вже існує в проєкті.

### 4. Список техніки, яка виконує цю послугу
Потрібно показати реальну техніку з каталогу.

Для кожної одиниці можна показати:
- фото;
- назву;
- короткі характеристики;
- кнопку переходу на сторінку техніки.

### 5. CTA-блок
- залишити заявку;
- отримати консультацію;
- підібрати техніку під задачу.

### 6. Додаткові блоки
За бажанням можна додати:
- FAQ;
- етапи роботи;
- пов’язані послуги;
- переваги компанії.

---

## Етап 6. Оновити сторінку конкретної техніки
На сторінці кожної одиниці техніки потрібно додати новий блок, наприклад:

- **«Види робіт, які може виконувати ця техніка»**
- або **«Послуги, для яких підходить ця техніка»**

### Вимоги до блоку
1. Блок має з’являтися лише тоді, коли для техніки є хоча б одна пов’язана послуга.
2. Блок має виглядати як окремий логічний секшен сторінки.
3. Послуги в цьому блоці можуть бути у вигляді:
   - компактних карток;
   - списку;
   - рядка карток.

### Для кожної пов’язаної послуги показати
- назву послуги / виду робіт;
- короткий опис або 1 короткий підзаголовок;
- кнопку **«Детальніше»**;
- кнопка повинна вести на сторінку відповідної послуги.

### Приклад UX-логіки
Якщо користувач відкрив сторінку екскаватора, він може побачити:
- Копання траншеї — кнопка «Детальніше»;
- Риття котлованів — кнопка «Детальніше»;
- Планування ділянки — кнопка «Детальніше».

Після кліку користувач переходить на сторінку відповідної послуги.

---

## Етап 7. Перевикористання існуючих компонентів
1. Якщо на сайті вже є готові картки, секції, кнопки або grid-компоненти — перевикористай їх.
2. Не створюй новий стиль без потреби.
3. Нові секції мають органічно виглядати як частина існуючого сайту.

---

## Етап 8. SEO-оптимізація
Потрібно зробити SEO-дружню структуру.

### Для сторінок послуг
1. Унікальний `title`.
2. Унікальний `meta description`.
3. Один `h1`.
4. Зрозумілі `slug` у URL.
5. Внутрішня перелінковка:
   - список послуг → сторінка послуги;
   - сторінка послуги → техніка;
   - сторінка техніки → сторінка послуги.

### Додатково
6. Якщо є breadcrumbs — додати їх.
7. Контент не повинен бути надто коротким і шаблонним.
8. Блок послуг на сторінці техніки теж має покращувати внутрішню перелінковку.

---

## Етап 9. UI/UX вимоги
1. Дизайн має відповідати стилю сайту.
2. Картки послуг повинні бути сучасними, чистими і читабельними.
3. Блок послуг на сторінці техніки не повинен виглядати випадковим або перевантаженим.
4. На мобільній версії:
   - картки мають коректно перебудовуватися;
   - кнопки мають бути зручними для натискання;
   - текст не повинен ламати сітку.
5. На сторінці послуги й сторінці техніки має бути чітка візуальна ієрархія.
6. Не допускати горизонтального скролу, зламаної сітки або надто високих карток без потреби.

---

## Етап 10. Технічна реалізація
Потрібно:
1. Додати маршрути для списку послуг і окремих сторінок послуг.
2. Створити джерело даних послуг.
3. Реалізувати логіку зв’язку через `relatedEquipment`.
4. Створити сторінку списку послуг.
5. Створити шаблон / компонент сторінки окремої послуги.
6. Оновити шаблон / компонент сторінки окремої техніки.
7. Додати блок пов’язаних послуг на сторінку техніки.
8. Реалізувати переходи:
   - послуга → техніка;
   - техніка → послуга.
9. Підключити SEO-метадані.
10. Переконатися, що немає помилок у консолі й битих маршрутів.

---

## Етап 11. Перевірка результату
Перед завершенням роботи потрібно перевірити все функціонально, візуально та логічно.

### Функціонально
- чи є окрема сторінка послуг;
- чи відкриваються сторінки кожної послуги;
- чи відображається релевантна техніка на сторінці послуги;
- чи відображаються релевантні види робіт на сторінці техніки;
- чи працює кнопка **«Детальніше»** на сторінці техніки;
- чи веде вона саме на правильну сторінку послуги;
- чи працюють переходи на сторінки техніки.

### Візуально
- чи гарно виглядає список послуг;
- чи охайно виглядає секція послуг на сторінці техніки;
- чи немає зламаної верстки;
- чи немає надто довгих текстів;
- чи немає стрибків layout.

### Логічно
- чи всі послуги відповідають реальній техніці;
- чи немає вигаданих машин;
- чи немає вигаданих зв’язків;
- чи немає сторінок послуг без відповідної техніки, якщо це не передбачено логікою.

---

# Очікуваний результат
У результаті має бути реалізована цілісна система:

1. **Сторінка послуг** зі списком карток.
2. **Окремі сторінки послуг** з описом, ціною / розрахунком і технікою.
3. **Оновлені сторінки техніки**, де є блок із видами робіт / послугами, які може виконувати ця техніка.
4. Повна внутрішня перелінковка між технікою і послугами.
5. Рішення повинно бути масштабованим: нову послугу можна легко додати через дані, без повного переписування логіки.

---

# Додаткові вимоги до агента
1. Спочатку проаналізуй фактичні дані техніки.
2. Не вигадуй техніку, категорії або ціни без опори на реальні дані.
3. Використовуй масштабований підхід.
4. Якщо є готові компоненти — перевикористовуй їх.
5. Якщо даних бракує, заклади гнучку структуру для подальшого наповнення.
6. Зроби так, щоб новий блок на сторінці техніки був інтегрований акуратно і логічно.
7. Після реалізації перевір, щоб усе реально працювало, а не було тільки в коді формально.

# Коротка інструкція агенту
1. Проаналізуй, яка техніка реально є на сайті.
2. Сформуй перелік валідних послуг на основі цієї техніки.
3. Створи структуру даних послуг.
4. Реалізуй сторінку списку послуг у форматі карток.
5. Реалізуй окремі сторінки для кожної послуги.
6. Прив’яжи до кожної послуги реальну техніку з каталогу.
7. Додай на сторінку кожної техніки блок із видами робіт, які вона може виконувати.
8. Для кожного виду робіт додай кнопку **«Детальніше»**, що веде на сторінку відповідної послуги.
9. Додай внутрішню перелінковку, SEO та перевір адаптивність.
10. Переконайся, що весь ланцюжок працює коректно: **послуга ↔ техніка**.

# Етап 12. Керування послугами через адмін-панель

## Загальна ціль
Реалізувати повноцінний CRUD для послуг в адмін-панелі, щоб адміністратор міг додавати, редагувати та видаляти послуги без зміни коду.

## Підготовка бази даних

### 1. Створити Prisma-модель `Service`
Потрібно перенести послуги зі статичних даних (frontend) у PostgreSQL.

Модель має містити:
- `id` — унікальний ідентифікатор (cuid)
- `slug` — URL-friendly текст (unique)
- `title` — назва послуги
- `shortDescription` — короткий опис для картки
- `fullDescription` — повний опис для сторінки послуги
- `image` — URL зображення
- `priceInfo` — текст вартості
- `pricingType` — enum: `fixed_from`, `hourly_from`, `calculator`, `custom`
- `features` — масив рядків (переваги послуги)
- `seoTitle` — SEO заголовок
- `seoDescription` — SEO опис
- `isActive` — чи показувати послугу на сайті (boolean, default true)
- `sortOrder` — порядок сортування (int, default 0)
- `createdAt`, `updatedAt`

### 2. Створити зв'язок Service ↔ EquipmentType
Додати join-таблицю або масив enum-значень `relatedEquipmentTypes`, щоб кожна послуга була прив'язана до типів техніки.

### 3. Міграція
- Створити Prisma-міграцію.
- Написати seed-скрипт, який перенесе існуючі статичні послуги в базу.

## Backend API

### 4. Створити серверні маршрути
- `GET /api/services` — публічний, повертає всі активні послуги (isActive: true), відсортовані за sortOrder.
- `GET /api/services/:slug` — публічний, повертає одну послугу за slug.
- `GET /api/admin/services` — авторизований, повертає всі послуги (включно з неактивними).
- `POST /api/admin/services` — авторизований, створити нову послугу.
- `PUT /api/admin/services/:id` — авторизований, оновити послугу.
- `DELETE /api/admin/services/:id` — авторизований, видалити послугу.

### 5. Валідація через Zod
- Додати Zod-схеми для створення та оновлення послуг.
- Валідувати slug (унікальність), title, pricingType enum тощо.

## Frontend адмін-панель

### 6. Створити сторінку `AdminServicesPage`
- Таблиця / список усіх послуг (назва, slug, isActive, кількість пов'язаних типів техніки).
- Кнопки: «Додати послугу», «Редагувати», «Видалити».
- Фільтр за статусом (активні / неактивні / всі).
- Пагінація (якщо послуг більше 10).

### 7. Створити форму додавання / редагування послуги
Форма має містити поля:
- Назва (`title`)
- Slug (автогенерація з title, з можливістю ручного редагування)
- Короткий опис (`shortDescription`)
- Повний опис (`fullDescription`) — textarea
- URL зображення (`image`) — або завантаження через upload
- Текст вартості (`priceInfo`)
- Тип ціноутворення (`pricingType`) — select з enum
- Переваги (`features`) — динамічний список рядків (додати/видалити)
- Пов'язані типи техніки (`relatedEquipmentTypes`) — мультиселект з чекбоксами
- SEO заголовок і опис
- Активність (`isActive`) — toggle
- Порядок сортування (`sortOrder`)

### 8. Додати маршрут і пункт меню в адмінку
- Додати маршрут `/admin/services` в `App.tsx`.
- Додати пункт «Послуги» в sidebar (`AdminLayout.tsx`).

## Перехід з статичних даних на API

### 9. Оновити фронтенд
- Замінити імпорт статичних даних з `services.ts` на виклики API (`GET /api/services`, `GET /api/services/:slug`).
- Створити `services.service.ts` (аналогічно до `equipment.service.ts`).
- Забезпечити зворотну сумісність: якщо API недоступний — fallback на статичні дані.

### 10. Перевірка
- Адмін може створити нову послугу → вона з'являється на `/services`.
- Адмін може деактивувати послугу → вона зникає з публічної частини.
- Адмін може змінити `relatedEquipmentTypes` → зв'язки на сторінці техніки оновлюються.
- Видалення послуги не ламає UI.

## Важливо
- Цей етап можна реалізувати **після** завершення етапів 1–11, коли статичні послуги вже працюють.
- Структура даних послуг (етап 2) має бути спроєктована з урахуванням майбутньої міграції в базу даних.
- Поля типу `Service` мають збігатися між статичним файлом і майбутньою Prisma-моделлю.
