# Інструкція для Claude Opus: поетапний рефакторинг адмін-панелі TechnoRent

## Роль
Ти працюєш як уважний frontend/UI/UX агент-рефакторинг для існуючої адмін-панелі сервісу оренди техніки. Твоя задача — **поетапно** покращити зручність адмінки без ламання поточного функціоналу.

Стек: **React + TypeScript + Tailwind CSS**.

Основний пріоритет: **зручність щоденної роботи адміністратора**, а не декоративність.

---

## Головна ціль
Потрібно переробити адмін-панель так, щоб вона стала:
- компактнішою
- зрозумілішою
- швидшою для роботи
- більш послідовною в UI
- безпечнішою щодо destructive actions
- готовою до масштабування

При цьому потрібно:
- не ламати існуючу бізнес-логіку
- не видаляти працюючий функціонал без причини
- не змінювати різко бренд-стиль
- зберегти фірмову схему: **темний інтерфейс + жовтий акцент**
- рухатися лише маленькими завершеними етапами

---

## Правила роботи

### 1. Працюй тільки поетапно
Не намагайся переписати всю адмінку за один раз.
Кожен етап має бути окремим, завершеним і перевіреним.

### 2. Перед кожним етапом обов’язково роби короткий план
Перед внесенням змін вкажи:
1. що саме будеш змінювати
2. які файли будуть змінені
3. які компоненти будуть створені або оновлені

### 3. Після кожного етапу обов’язково роби звіт
Після завершення етапу вкажи:
1. що реалізовано
2. які файли змінено
3. чи є тимчасові обмеження
4. що логічно робити наступним етапом

### 4. Не вигадуй нову бізнес-логіку без потреби
Якщо даних або API для певної функції поки немає:
- використовуй тимчасовий UI
- не ламай існуючу структуру
- залиш місце для майбутньої інтеграції

### 5. UX важливіший за декоративність
Це адмінка, не лендинг. Потрібно оптимізувати:
- швидкість сканування даних
- доступність основних дій
- компактність інтерфейсу
- зрозумілість статусів
- простоту керування заявками, технікою і зайнятістю

### 6. Не переходь до наступного етапу, поки не завершено поточний
Після кожного етапу перевір:
- чи немає помилок збірки
- чи не зламалась верстка
- чи не з’явився випадковий горизонтальний скрол
- чи збережено коректну поведінку на desktop і mobile

---

## Яким має бути фінальний результат
Після завершення всіх етапів адмін-панель має:
- мати окрему сторінку **Огляд / Dashboard**
- мати компактніші й зручніші таблиці
- мати єдині статуси у вигляді бейджів
- мати покращені пошук і фільтри
- мати менш агресивні destructive actions
- мати зручнішу сторінку заявок для швидкої обробки
- мати більш практичний календар зайнятості
- мати єдиний візуальний стандарт для header, card, buttons, filters, tables, detail panels

# Етап 0. Аудит структури адмін-панелі

## Ціль
Спочатку проаналізувати поточну структуру інтерфейсу, перш ніж щось глобально змінювати.

## Що потрібно зробити
1. Знайти:
- layout адмінки
- sidebar
- сторінку техніки
- сторінку заявок
- сторінку календаря / зайнятості
- загальні UI-компоненти: button, input, card, table, badge, modal, select

2. Визначити:
- які стилі дублюються вручну
- чи є shared layout-контейнер
- чи є часткова дизайн-система або спільні UI-компоненти
- де використовуються кнопки `Видалити`, `Редагувати`, статуси, таблиці

3. Після аналізу:
- коротко описати, які файли/компоненти потрібно створити або рефакторити першими
- не робити на цьому етапі великого переписування
- якщо потрібно, підготувати основу для reusable-компонентів

## Очікуваний результат
Короткий аудит структури і план першого етапу рефакторингу.

# Етап 1. Базова UI-система для адмінки

## Ціль
Уніфікувати базові елементи інтерфейсу, щоб не дублювати стилі вручну.

## Потрібно створити або привести до єдиного стандарту

### 1. `AdminPageHeader`
Компонент для верхньої частини сторінки:
- заголовок
- підзаголовок
- зона праворуч для основної дії

### 2. `StatBadge`
Єдиний компонент статусів.

Підтримати статуси:
- `new` / Новий
- `in_progress` / В обробці
- `confirmed` / Підтверджено
- `completed` / Завершено
- `cancelled` / Скасовано
- `available` / Вільно
- `busy` / Зайнято
- `maintenance` / На обслуговуванні
- `inactive` / Неактивна

Вимоги:
- єдина висота
- однакові падінги
- фон + колір тексту залежно від статусу
- компактний бейдж-стиль

### 3. Єдина система кнопок
Уніфікувати стилі кнопок:
- `primary`
- `secondary`
- `ghost`
- `danger`

### 4. `AdminFilterBar`
Уніфікований блок для:
- пошуку
- select-фільтрів
- кнопок типу `Лише нові`, `Експорт`, `Оновити`

### 5. `AdminCard`
Єдина картка для:
- правих панелей
- статистики
- деталей
- вторинних інформаційних блоків

### 6. `AdminTable`
Не обов’язково повністю абстрактний компонент, але таблиці треба привести до одного стилю:
- компактніші рядки
- однакові відступи
- hover state
- selected state
- sticky header, якщо можливо

## Вимоги до стилю
- трохи зменшити надмірні відступи
- зробити інтерфейс компактнішим
- зберегти сучасний чистий вигляд
- не перевантажувати декоративністю
- жовтий лишити primary accent
- danger-стилі не повинні домінувати

## Очікуваний результат
Після етапу всі наступні сторінки мають будуватися з узгоджених компонентів.

# Етап 2. Переробка загального layout адмінки

## Ціль
Покращити каркас адмінки, щоб він був зручнішим і менш громіздким.

## Що змінити

### Sidebar
- зробити трохи компактнішим
- зменшити висоту верхнього блоку з логотипом
- зробити активний пункт меню більш виразним:
  - жовтий фон
  - і/або ліва акцентна смуга
- зробити нижній блок користувача компактнішим
- кнопку `Вийти` зробити менш громіздкою

### Меню
Додати пункт:
- `Огляд`

### Main content area
Стандартизувати:
- однакові горизонтальні відступи
- однакову максимальну ширину
- однаковий вертикальний ритм між header, filters, content

### Адаптивність
Перевірити:
- щоб layout не ламався на менших екранах
- щоб mobile/tablet версія не деградувала

## Очікуваний результат
Каркас адмінки стає більш професійним і придатним до подальшого росту.

# Етап 3. Нова сторінка “Огляд” / Dashboard

## Ціль
Створити головну сторінку, яка дає адміну загальну картину за 3–5 секунд.

## Структура сторінки

### Верхній блок
- заголовок `Огляд`
- короткий опис
- можлива кнопка швидкої дії справа

### Ряд статистичних карток
- Нові заявки
- Активні оренди
- Вільна техніка
- На обслуговуванні

### Основні інформаційні блоки
- Останні заявки
- Найближчі завершення оренди
- Попередження / проблемні місця, якщо дані доступні

### Блок швидких дій
- Додати техніку
- Додати зайнятість
- Перейти до заявок
- Експортувати заявки

## Якщо бекенд ще не дає всі дані
Побудувати сторінку так, щоб:
- легко підключити реальні дані пізніше
- тимчасово можна було використовувати mock або derived state

## Очікуваний результат
Адмін відразу бачить головні цифри і пріоритетні задачі.

---

# Етап 4. Переробка сторінки “Заявки”

## Ціль
Зробити сторінку заявок максимально швидкою і зручною для обробки.

## Що потрібно змінити

### 4.1 Верх сторінки
Використати `AdminPageHeader`:
- заголовок
- підзаголовок
- дії справа: `Оновити`, `Експорт`

### 4.2 Фільтри
Привести до єдиного стилю:
- пошук по імені/телефону
- фільтр по статусу
- фільтр по техніці, якщо дані доступні
- опціонально кнопка `Лише нові`

### 4.3 Таблиця заявок
Потрібно:
- зробити рядки компактнішими
- чітко виділити selected row
- прибрати зайве візуальне навантаження
- іконку видалення з рядка прибрати або сховати в меню дій
- destructive action не має бути найпомітнішим елементом

### 4.4 Статуси
Замінити кольоровий текст на `StatBadge`.

### 4.5 Права панель “Деталі заявки”
Зробити її більш структурованою.

Показувати:
- статус
- клієнт
- телефон
- техніка
- дата заявки
- коментар, якщо є
- додаткові поля, якщо є в моделі

### 4.6 Дії в деталях заявки
Додати або привести до ладу дії:
- Підтвердити
- Позначити як в обробці
- Завершити
- Скасувати
- Видалити або Архівувати — тільки як другорядну дію

### 4.7 Акцент на нових заявках
Нові заявки мають бути помітнішими:
- бейдж
- сильніший візуальний акцент у рядку, але без перевантаження
- лічильник у меню, якщо логіка це дозволяє

## Очікуваний результат
Адміністратор може швидко:
- знайти заявку
- відкрити її
- змінити статус
- побачити деталі
- не плутатися в списку

# Етап 5. Переробка сторінки “Техніка”

## Ціль
Зробити сторінку техніки більш інформативною і готовою до масштабування.

## Що потрібно зробити

### 5.1 Верх сторінки
Стандартизувати header:
- заголовок
- опис
- кнопка `Додати нову техніку`

### 5.2 Filter bar
Додати:
- пошук по назві
- фільтр по статусу
- фільтр по категорії, якщо є категорії
- сортування по ціні / назві / статусу, якщо неважко реалізувати

### 5.3 Таблиця техніки
Бажані колонки:
- назва
- категорія або тип, якщо є
- ціна/год
- статус
- найближча зайнятість або коротка інформація про доступність, якщо це можливо
- дії

### 5.4 Дії
Не показувати великі кнопки `Редагувати` і `Видалити` прямо в кожному рядку, якщо це перевантажує інтерфейс.

Краще:
- компактна кнопка `Дії`
- dropdown / popover:
  - Редагувати
  - Архівувати
  - Видалити

### 5.5 Статуси
Усі статуси техніки перевести в `StatBadge`.

### 5.6 Архівування замість повного видалення
Якщо поточна логіка дозволяє, віддати перевагу архівуванню.

## Очікуваний результат
Сторінка техніки стає структурованішою, менш шумною і готовою до росту кількості позицій.

---

# Етап 6. Переробка сторінки “Календар зайнятості”

## Ціль
Зробити календар реальним робочим інструментом, а не лише візуальним блоком.

## Що потрібно зробити

### 6.1 Верх сторінки
Стандартизувати:
- заголовок
- підзаголовок
- дії справа за потреби

### 6.2 Покращити календар
Потрібно, щоб у кожному дні було легко зчитувати інформацію:
- якщо день вільний — це видно одразу
- якщо є кілька зайнятостей — показати короткий summary
- оренда / бронювання / сервіс мають відрізнятися кольором або бейджами

### 6.3 Права панель
Зробити список записів зручнішим:
- фільтр по техніці
- список активних записів для вибраного дня або періоду
- в кожному записі:
  - назва техніки
  - дати
  - статус
  - дії

### 6.4 Кнопка `Додати зайнятість`
Зробити її добре видимою, але без перевантаження інтерфейсу.

### 6.5 Легенда
Створити компактну і зрозумілу легенду:
- бронювання
- оренда
- техобслуговування
- вільно

## Бажано, якщо це не надто складно
Підготувати архітектуру під майбутні режими перегляду:
- `Місяць`
- `Таймлайн`

Але не обов’язково повністю реалізовувати timeline на цьому етапі.

## Очікуваний результат
Календар швидко показує завантаженість і дозволяє керувати записами без плутанини.

---

# Етап 7. Безпечні destructive actions

## Ціль
Зменшити ризик випадкового видалення.

## Що потрібно зробити
1. У всіх місцях, де є `Видалити`:
- зробити цю дію менш акцентною
- не використовувати її як primary action

2. Додати confirm modal:
- чіткий текст
- назва об’єкта, який видаляється
- confirm-кнопка
- cancel-кнопка

3. Де можливо:
- замінити `Видалити` на `Архівувати`

## Очікуваний результат
Менше ризику помилок і професійніша поведінка системи.

Інтерфейс виглядає цілісним, спокійним і зручним для реальної роботи.

# Технічні вимоги

## По стилях
- використовувати Tailwind CSS
- мінімізувати хаотичні дубльовані класи
- повторювані патерни виносити в reusable-компоненти, якщо це дає реальну користь
- зберегти фірмовий характер: темний + жовтий

## По React / TypeScript
- не створювати надто складні абстракції без користі
- компоненти мають бути зрозумілими
- типізація має бути чистою
- не змішувати layout, логіку, таблиці і form actions в одному громіздкому компоненті, якщо це можна розділити

## По UX
- адмінка має бути компактнішою за лендинг
- на екрані має поміщатися більше корисної інформації
- найважливіші дії мають бути найзручнішими
- destructive actions мають бути візуально другорядними

# UX-принципи, яких потрібно дотримуватися
- Не роби інтерфейс надто повітряним
